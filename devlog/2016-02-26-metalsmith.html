Harp

Currently I'm using Harp to generate my site. I have two main issues with Harp:

  • It stores metadata in a per-directory data.json, rather than in a file header like most tools. This is inconvenient, since it requires editing two files to author a post. It also defies a strong convention (Jekyll and most other static site tools).
  • It takes a strong stance on limiting functionality, self-consciously omitting any sort of extension mechanism. Part of me likes this, but over the years I have had to work around or accept many limitations.

Metalsmith

I'm currently evaluating Metalsmith, another tool for building static websites.

Where

pipeline runner for compiling websites. It takes a source directory containing files, creates a map of each file in the directory with metadata parsed, and then a series of plugins which transform the map to produces the output.

It's kind of an anti Harp.

  • Harp is meant to be a hands-off "appliance" which turns a directory full of posts, metadata files, and templates into a site
  • Metalsmith

The value proposition:

  • You configure your site processor using a file metalsmith.json.

  • The authors claim "everything is a plugin", which is almost true. The only functionality in the core is:

    • Recursively scan a directory for regular files, and create a map {path: object}, where object includes metadata extracted from the file's YAML (or JSON, etc) header as well as the data stream.
    • Call each plugin in order with the map.
  • The plugin interface is extremely simple, essentially a single function which is passed the mutable map described above.

Resources: https://github.com/metalsmith/awesome-metalsmith
